# 市場インサイトと設計への反映（2026-06）

ユーザー実施の市場調査（競合・標準機能・ペインポイント・AI青写真）を、
Sourirette の設計判断に翻訳したもの。原文の要旨 → 「うちはどうするか」の形式で整理する。

> 注: 原調査内の数値（シェア22%/17%、診断ばらつき35%削減等）は出典未確認のため、
> 対外資料に使う際は一次ソースの裏取りを行うこと。

---

## 1. 競合環境の要点

| 勢力 | 代表 | 強み | 弱み |
| --- | --- | --- | --- |
| オンプレ2強 | WiseStaff（ノーザ）/ palette（ミック） | 数十年分の算定ルール対応力、24/365サポート、安定稼働 | 高額初期費用、UIの古さ、改定対応が院側負担、リース縛り |
| クラウド新興 | Dentis（メドレー）/ POWER5G 等 | 初期費用低、改定自動適用、訪問先アクセス | 算定の深さ・実績はオンプレ勢に劣後 |

**含意**: 勝負所は「クラウドの身軽さ × オンプレ級の算定の深さ × AIネイティブ」。
算定エンジンの精度はオンプレ2強が数十年かけた領域であり、Phase 2 の
「既存レセコンとの算定突合（差分ゼロ）」を品質戦略の中心に置く方針は変えない。

## 2. 調査が裏付けた Phase 0 の設計判断（自信を持って続行）

1. **代行入力と医師確定の分離**（調査 §4.3）
   カルテ記載・請求入力は歯科医師の義務であり、衛生士の単独入力は違法。
   既存システムはこのワークフロー分離が弱い。
   → うちは初日から `authored_by`（記載者）/`finalized_by`（確定者・歯科医師のみ）を
   分離し、追記専用＋確定操作で設計済み。**これは差別化機能として前面に出す。**
2. **音声→SOAP自動生成**（§6.2）: dentia 統合の路線そのもの。入力80%削減の市場期待。
3. **多段階のリアルタイム算定チェック**（§3.3）: 入力時・会計時・月次の3段チェックは
   実装済みの設計（エラー=ブロック/警告=要確認）と一致。

## 3. 新たにロードマップへ追加するもの

### 3.1 P検（歯周ポケット検査）モジュール — Phase 1 に昇格
- 全システム標準。数値（PD）・BOP・動揺度から病態（P1〜P3）を自動判定し
  Sc/SRP 等の適応処置を提示するのが業界標準。
- **AIならではの解き方**: フットスイッチの代わりに、検査中の読み上げ音声
  （「4・6・3、BOPプラス…」）を文字起こしして自動入力する
  **ハンズフリー音声P検**。dentia の音声基盤がそのまま使え、専用デバイス不要。

### 3.2 データポータビリティ — 思想として全フェーズに
- 移行障壁とベンダーロックインが業界最大の不満（§4.2）。
- 攻め: **移行AI** — 他社CSV/UKE/紙カルテスキャン/PDF を AI で構造化して取込む
  「引っ越しウィザード」。移行困難が常識の市場では入口の武器になる。
- 守り: **全データの標準形式エクスポートを公式機能として保証**（カルテ全文・画像・
  算定履歴を含む）。「Sourirette はロックインしない」を契約とプロダクトで明文化。
  ※ 電子カルテ情報共有サービス（FHIR）対応とも整合。

### 3.3 オフライン耐性（ローカルファースト） — Phase 1 でアーキテクチャ決定
- 訪問歯科の急増に対し、通信断でも入力継続→回復時同期の「ローカルカルテ」が要件。
- これは後付け不可能な設計判断（同期・競合解決・追記専用ログの整合）のため、
  **Phase 1 のデータ層設計時に決める**。追記専用イベントログ中心の設計は
  オフライン同期と相性が良い（イベントの追記はマージ可能）。

### 3.4 電子処方箋・次世代基盤 — regulatory-checklist に追加
- 2026年度の電子処方箋管理サービス・電子カルテ情報共有サービスの経過措置期限、
  YJコード廃止等のマスタ仕様変更（§5.4）。オン資と連動したワンクリック署名
  （ローカル/リモート署名）を Phase 3〜4 の必須要件に組み込む。

## 4. AI 5モジュール青写真への対応方針

| 調査の提案 | 採否 | 理由・時期 |
| --- | --- | --- |
| §6.2 会話型カルテ自動生成 | **採用・本線**（Phase 1） | dentia 統合そのもの。実証済みの中核差別化 |
| §6.4 算定AI・返戻予測 | **段階採用**（Phase 2-3） | まずルールベース（実装済基盤）で確実に。返戻「予測」は全国レセプトデータが必要でコールドスタート問題あり。自院の返戻実績学習→共有データと段階を踏む |
| §6.5 経営分析・マーケAI | 採用（Phase 4） | 技術的に低リスク・データが貯まってから効く |
| §6.1 画像解析AI（病変検出） | **慎重**（Phase 5+/提携） | 病変検出は SaMD（プログラム医療機器）該当の可能性が高く、薬機法の承認/認証が必要になり得る。「AIは下書きまで・確定は医師」原則の外側。自社開発でなく承認済みプロダクトとの連携を基本線に |
| §6.3 遠隔モニタリング | 後回し（矯正特化） | 保険診療の本線が立ってから。API連携で足りる |

## 5. UI/UX 原則への追記（§4.1 への回答）

- **クリック数を仕様にする**: 主要動線（よくある処置の入力完了まで）の目標タップ数を
  画面仕様に明記し、リリース毎に計測する（例: 再診→Sc 算定完了まで5タップ以内）。
- 学習コスト: 新人スタッフが研修なしで受付業務を回せることを Phase 1 の受入基準にする。
- 「数秒の遅延の蓄積」が経営に効く（§4.1）→ チェアサイド画面は体感速度を最優先
  （楽観的UI更新・先読み）。
