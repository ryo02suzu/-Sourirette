# Sourirette マスターロードマップ — 土台から「AIを駆使した日本最高のレセコン一体型電子カルテ」まで

> **この文書は何か**: Sourirette（歯科レセコン一体型電子カルテ、TypeScript、純粋コア＋199テスト、CSV由来の確定算定エンジン、AIは未実装、臨床的正しさは未検証）を、商用で実医院が使える状態を経て、AIを駆使した日本最高のプロダクトにするための、最初から最後までの実装・事業ロードマップ。
> **使い方**: フェーズ順に進める。各フェーズに「目的／終了ゲート／実装タスク／DoD（完了の定義）／KPI／Claude Code着手プロンプト／撤退・ピボット判断」を置いた。法規制・事業・AI基盤は単独フェーズにせず**横断トラック**として全フェーズ並行で進める（後付けできないため）。
> **前提となる正直な現在地**: いまは「モダンに作り直したルールベースの土台」。AIはゼロ。算定エンジンは7ルールのみで実レセプトの多くをまだ正しく計算できない。臨床的正しさは未証明。**ここがスタートライン。**

-----

## 0. 全体像

### 0.1 北極星（North Star）

**歯科医が「書く・喋る」だけで、正しい算定・正しいレセプト・正しい記録が、根拠つきで自動的に立つ。算定もれと査定を最小化し、入力とレセ作業の時間を激減させる。最終判断は常に歯科医師。**

### 0.2 絶対に動かさない設計の不変条件

1. **数字・確定はコード、判断・生成はAI**（金額計算をLLMにさせない）。
1. **診断はさせない**（SaMD規制回避。AIは算定・記録の支援に限定）。
1. **すべて根拠つき（RAG）**（算定可否・点検は通知/事例の原文を提示）。
1. **人間が最終確定**（AIは提案・下書き、非ブロック）。
1. **算定忘れは自動付与しない・ブロックしない**（不正請求の回避）。
1. **公式データを正とする**（点数・コード・ルールはハードコードせず取込、世代管理で改定吸収）。

### 0.3 フェーズ全体図（製品ビルド）

```
現在地（土台・199テスト・AIゼロ・未検証）
  │
[P0] 正しさの証明 ── 実医院突合で既存レセコンと完全一致      ← 存在の前提（ここが緑になるまで先へ進まない）
  │
[P1] 1医院実運用MVP ── 実家クリニックで実請求（影運用→本番）  ← 商用の最低条件を1医院で満たす
  │
[P2] AIネイティブ化 ── カルテ/音声→算定起票・査定説明・詳記下書き ← 差別化の心臓
  │
[P3] 商用化・SaaS・認証 ── 外部医院へ有償提供（β→一般）       ← 法規制・認証・販売
  │
[P4] スケール・日本最高 ── 横展開・情報共有・高度AI・カテゴリ確立
```

### 0.4 横断トラック（全フェーズ並行・後付け不可）

- **トラックA：法規制・安全管理**（3省2／個情法／オンライン資格確認／電子処方箋／HPKI／SaMD境界／SaaS認証）
- **トラックB：データ運用**（改定・月次追補の自動追従／マスタ拡充／回帰ロック）
- **トラックC：事業**（エンティティ／資金／価格／GTM／サポート／学生身分と2028免許の整合）
- **トラックD：AI基盤・評価**（RAG基盤／eval／ガードレール）

### 0.5 illustrativeなタイムライン（目安・断定しない）

免許取得（2028）を基準にした相対目安。**コード作業はClaude Codeで加速できるが、突合・臨床検証・法規制・認証は実世界の遅延があり短縮できない。**日付は約束ではなくゲート（達成条件）で進む。

# Phase 0 — 正しさの証明（突合ゲート）

> **これが全ての前提。算定が1円でもズレるレセコンは、機能がいくつあっても現場で使えない。実家クリニックの実レセプトで既存レセコンと完全一致を取るまで、P1以降に進まない。**

### 目的

Souriretteの確定エンジンが、実際の歯科レセプトを既存レセコンと**1行・1点・1回・算定日まで完全一致**で計算できることを証明する。同時に、突合で露見する算定エンジンの欠落（投薬・材料・公費・通則加算など）を露見順に埋める。

### 終了ゲート（Exit Gate）

- 実家クリニックの **直近Nヶ月（できれば6〜12ヶ月）・全レセプト**で、エンジン出力が提出済みレセと完全一致。
- 不一致が残る場合は、すべて「既知の許容差」として理由を明文化（黙って残さない）。

|-----|---------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------|

### KPI

- 突合一致率（行/点/回数/算定日それぞれ）→ **100%が目標**
- 類型別の未一致件数（社保/国保/公費/高齢/小児/訪問）
- 露見→修正のリードタイム

### Claude Code 着手プロンプト

```
Souriretteに実医院レセプト突合ハーネスを実装したい。
- 入力: 既存レセコンが出力した実UKEファイル（匿名化済み）。これをパースして ProcessReceiptInput を復元する逆変換を作る。
- 実行: 復元した入力を現エンジン(processReceipt)で再計算。
- 比較: 元のUKEとの差分を「請求行・点数・回数・算定日・包括除外・摘要コメント」の粒度で取り、不一致を構造化レポート(JSON+人間可読)に出す。npm run verify:reconcile で動くCLIに。
- 個人情報は突合前に匿名化する前処理を必ず通す（PII除去関数を分離）。
- まず1レセプトで往復が一致することを確認するユニットテストから。次に複数月のディレクトリを一括突合し、類型別の一致率サマリを出す。
不一致が出たら、その原因（未実装の算定ルール）を特定しやすいよう、差分に「どのルールが関与したか」のトレースを付けて。
```

### 撤退・ピボット判断

- 地道な突合の末、エンジンが実レセと一致せず、かつ原因が「取得不能なデータor解釈が必要で機械化不能」に集中するなら、**全自動算定の野望を縮小し、AI点検支援（既存レセコンの隣に立つ）にピボット**を検討。
- ただし通常は「未実装ルールの追加」で潰せる。安易に諦めない。

# Phase 1 — 1医院実運用MVP（実家クリニックでドッグフード）

> **唯一にして最強の検証資産＝実家クリニック**を、Souriretteで実際に回す。いきなり本番に賭けず、**影運用（既存レセコンと二重記帳・毎月突合）→ 完全一致を継続 → 本番切替**の順で、クリニックの収益（返戻リスク）を守りながら進める。

### 目的

1医院（実家）が、受付→カルテ→算定→会計→レセプト提出までを日常的にSouriretteで完結できる、商用の最低条件を満たす状態。

### 終了ゲート（Exit Gate）

- 影運用で既存レセコンと**3ヶ月連続・差異ゼロ**。
- 本番切替後、**3ヶ月連続・返戻ゼロ**（or 返戻が出ても原因がSourirette起因でないと説明可能）。
- 受付・衛生士・歯科医がマニュアルなしで日次フローを回せる。

# Phase 2 — AIネイティブ化（差別化の心臓）

> ここで初めて「AIを駆使した」になる。**確定エンジンが正しい（P0/P1済）ことが絶対前提**。土台が正しくないとAIは「速く間違える」だけ。実家クリニックを実験場に、ガードレール内でAIを載せる。

### 目的

歯科医の入力・レセ作業を激減させ、算定もれ・査定を減らすAI機能群を、§0.2の不変条件を守って実装する。

### 終了ゲート（Exit Gate）

- 実家で、AI起票がレセ入力時間を**有意に削減**（目標値は計測して設定）。
- AI由来のハルシネーションによる誤確定が**ゼロ**（人間確定の前提で安全に運用できている）。
- すべてのAI出力に根拠（通知/事例の原文）が付く。

### 実装タスク

|#   |タスク                      |詳細 / DoD                                                                                                 |
|----|-------------------------|---------------------------------------------------------------------------------------------------------|
|P2-1|AI基盤（RAG）                |留意事項通知・審査情報提供事例・施設基準通知の全文を取り込み、検索（embedding+retrieval）基盤を構築。AIには「思い出させる」のでなく「検索して渡す」。DoD: 算定可否クエリで根拠原文が返る|
|P2-2|AI評価ハーネス（先に作る）           |起票精度・再現率、誤検知率、ハルシネーション検知、根拠付与率を測る eval。実家の実データでベンチ。DoD: AI変更ごとに自動評価が回る                                   |

|P2-4|**カルテ自由記述/音声 → 算定起票（本丸）**|「右下6 Per 感根処 根貼」等の自然文/音声→算定コード候補＋必要病名を構造化提案→確定エンジン(§2)で検算→人が確定。DoD: 自然文→算定候補が高精度、誤りは人が修正                 |
|P2-5|音声入力（チェアサイド）             |喋ってカルテ＋算定起票。キーボードを触れない現場価値。DoD: 音声→カルテ＋算定                                                                |

|P2-8|病名自動付与“提案”               |処置から必要病名を逆引き提案（提案のみ・自動付与しない）。DoD: 提案を人が承認して付与                                                            |
|P2-9|SaMD境界の判定・記録             |AI機能ごとに薬機法該当性を判定し記録。診断支援に踏み込まないことを担保。DoD: 各機能の該当性判定が文書化                                                  |

### KPI

- 起票精度・再現率
- レセ入力時間削減率
- 算定もれ検出数（収益）
- ハルシネーション発生数 → 0
- 根拠付与率 → 100%

### Claude Code 着手プロンプト

```
SouriretteにAI支援層を追加する。ただし不変条件を厳守:
(1)金額・確定計算はLLMにさせない（既存の確定エンジンが計算・検算）(2)診断はさせない(3)出力は必ず根拠(通知/事例原文)つき(4)提案/下書きのみで人間が最終確定(5)算定忘れは自動付与しない。

まずAI基盤から:
1. RAG: 留意事項通知・審査情報提供事例・施設基準通知の全文をチャンク化し検索可能に（embedding+retrieval）。根拠原文を返すretrieverを作る。
2. eval: 起票精度/再現率・誤検知率・ハルシネーション検知・根拠付与率を測る評価ハーネス（実データでベンチ、AI変更ごとにCI実行）。
3. ガードレール層: LLM出力を必ず確定エンジンで検算し、数字の食い違い・根拠なし・診断的表現を弾くフィルタ。

その上で本丸:
4. カルテ自由記述→算定起票: 自然文を解析して算定コード候補＋必要病名を構造化提案→確定エンジンで検算→UIで人が確定。LLMは抽出と提案のみ。
段階的に音声入力・査定リスク説明・症状詳記下書きへ拡張。各機能にSaMD該当性メモを残す。
```

### 撤退・ピボット判断

- 起票精度が実用に満たない／ハルシネーション制御が効かないなら、AIを**起票（生成）から点検（検知・説明）寄りに後退**（生成より検証の方が安全で価値が出る）。

### トラックC（事業・免許との整合）— ここを正直に設計する

- 免許取得（2028）前は、**請求主体になれるのは保険医（歯科医師）＝医院側**。Souriretteはソフト提供者であり、請求の最終責任は各医院の保険医が負う（プロダクトの位置づけ通り）。
- **学生という立場は、顧客発見・ヒアリングでは武器、直接の商用営業では弱点**になりやすい。初期は実家＋warm introのdesign partnerで実績を作り、本格的な外部販売は免許取得前後 or 営業/事業を担える共同創業者・法人体制を整えてからにする。
- 二者間マーケットプレイス的なコールドスタートには手を出さず、**B2Bの直販（医院単位）に集中**する。

### KPI

- 有償医院数、返戻率、解約率/NRR、サポート負荷、導入リードタイム、粗利。

### Claude Code 着手プロンプト

```
Souriretteを外部医院に提供できるSaaSにする。
1. マルチテナント化: 医院単位の完全データ分離、テナント識別をデータ層・API・認可に貫通。
2. 3省2ガイドライン準拠の技術要件: 監査ログの保全期間・改ざん検知、アクセス制御の強化、バックアップ/復旧、暗号化（保存時・通信時）。準拠項目をチェックリスト化してコードと対応づける。
3. オンライン資格確認・電子処方箋・電子カルテ情報共有サービスの連携IF（外部システム連携層を分離して実装）。
4. 管理機能: テナント管理・利用状況・課金連携・改定マスタの配信。
セキュリティと可用性を最優先。認証取得を見据え、設計判断を docs に記録する。
```

### 撤退・ピボット判断

- 認証・規制コストが事業性を圧迫するなら、**オンプレ提供 or 既存ベンダーへのOEM/技術提供**に形態転換を検討。
