# Sourirette システム仕様書

レセコン一体型 歯科電子カルテ（フルスクラッチ）

| 項目 | 内容 |
|---|---|
| システム名 | Sourirette（ソーリレット） |
| 種別 | レセプト電算一体型 歯科電子カルテ／算定支援システム |
| 対象 | 日本の保険診療を行う歯科医院（歯科・歯科口腔外科） |
| 準拠改定 | 令和8年度診療報酬改定（令和8年6月1日施行） |
| 文書バージョン | 1.0（2026-06-14 時点の実装に基づく） |
| 実装言語 | TypeScript（コアは外部ランタイム依存ゼロ）／UIは React + Vite |
| 検証 | ユニットテスト189件（`npm test`）すべてパス |

> **位置づけ（重要）**: 本書は「コード上で実装・動作確認済みの機能」の仕様である。
> 算定の臨床的正しさ（手計算・既存レセコンとの一致、確認試験合格、返戻ゼロ）は
> 実医院での突合検証（研鑽）を経て確定する。本書は実装仕様であって、商用請求の
> 正当性を保証するものではない。最終的な請求の妥当性は保険医（歯科医師）が負う。

## 1. システム概要

### 1.1 目的

歯科医院の日々の診療入力（カルテ）から、保険点数の算定・レセプト電算ファイル（UKE）の生成・
提出前点検・窓口会計までを一気通貫で行う。あわせて算定もれ・算定誤りを提出前に提示し、
返戻・査定を未然に減らす。電子カルテとしての法的要件（電子保存の3原則）の技術的中核も担う。

### 1.2 設計思想

1. **公式データを正とする**: 点数・コード・ルールはハードコードせず、支払基金等が公開する
   マスタ・電子点数表・告示通知を取り込んで使う。改定は世代管理（valid_from/valid_to）で吸収。
2. **エンジンは勝手に確定しない**: 算定結果・指摘は提示するが、最終確定は歯科医師の操作。
   ただし機械的に一意に決まる事項（包括除外・年齢/時間外加算）は自動適用する。
3. **誤検知ゼロ設計**: 正しいレセプトでは警告を出さない。警告は高精度な負の信号のみ。
4. **純粋なコア**: `src/billing` 配下は fs 非依存でブラウザでも動く。ファイル入出力は呼び出し側。

### 1.3 スコープ

| 対象 | 範囲 |
|---|---|
| 含む | 歯科の保険点数算定、UKE生成、提出前点検、窓口会計・高額療養費、電子カルテ保存中核、算定支援アラート |
| 含まない（コード外） | オンライン請求の申込・電子証明書・閉域網、実機の確認試験、改定追従の運用体制、DB永続化の本番構築 |

### 2.2 標準データフロー（1患者・1ヶ月）

```
カルテ入力（受診・処置コード・傷病名）
   │  ProcessReceiptInput
   ▼
loadOfficialEngine で構成したエンジンで受診ごとに算定
   │  実点数 + ルール発火（回数/背反/包括/部位/病名/加算）
   ▼
月次集約（複数受診→1レセプト・算定日マージ・診療識別グループ化）
   │  UkeReceipt
   ▼
UKEレコード組立 → Shift_JIS バイト列化（EOF付き）
   │
   ├─→ 提出前自己点検（L1/L2）→ submittable 判定
   ├─→ 算定支援アラート（error/warning/proposal）
   ├─→ 摘要欄コメント候補（別表Ⅰ）
   └─→ 会計（費用区分別・窓口負担・高額療養費）
   ▼
ProcessReceiptResult（UKE base64・点検結果・点数・アラート・会計）
```

## 4. 算定エンジン仕様

### 4.1 入力（ProcessReceiptInput 抜粋）

```ts
{
  facility: { payer, prefecture, facilityCode, facilityName, billingMonth, phone? },
  patient:  { birthDate: "YYYY-MM-DD", sex: "M"|"F" },
  name, kanaName?, chartNo?,
  scheme:   { kind: "medical"|..., beneficiary: "self"|"family"|... },
  insurer:  { insurerNo, symbol?, number },
  visits: [ { date, visitType: "first"|"followup", procedureCodes: string[], timeClass? } ],
  diagnoses: [ { diseaseCode, teeth?: FDI[], onsetDate } ],
  notifiedStandards?: string[],   // 届出済み施設基準
  acknowledgedAlerts?: string[],  // 既読アラート
  copay?: { copayRatio, category, isMultiple?, applyCapAtWindow? }
}
```

`timeClass`: `"regular"|"afterHours"|"holiday"|"midnight"`（時間外/休日/深夜加算の自動算定に使用）。

### 4.2 ルール群（評価順）

エンジンは小さなルールの集合。各ルールは適用期間を持ち、診療日で発火判定する。

| 順 | ルール | 種別 | 動作 |
|---|---|---|---|
| 1 | pricingRule | 点数付与 | 各診療行為コードを実マスタ点数で算定行に。マスタに無いコードは error（黙って落とさない） |
| 2 | createAdditionRule | 自動加算 | 初診料/再診料が算定済みのとき、患者年齢（6歳未満）と時間区分から乳幼児/時間外/休日/深夜加算をマスタ名称で解決し自動付与。二重・過剰算定しない |
| 3 | frequencyLimit | 回数制限 | 同月の算定回数が上限超過なら error（電子点数表 算定回数） |
| 4 | mutualExclusion | 併算定不可 | 同日/同月で背反する組合せなら error（電子点数表 背反） |
| 5 | inclusionGroup | 包括 | 子が親に包括される場合、子を**請求行から除外**し error 指摘（excludesFromBilling） |
| 6 | siteDiagnosis | 部位×病名突合 | 処置の歯番が傷病名部位に無ければ error。歯番病名がゼロのときはスキップ（誤検知防止） |
| 7 | diagnosisRequirement | 病名適応 | 禁忌病名（不適応）に該当すれば warning |

### 4.3 提出可否（submittable）

提出可能 = UKE形式点検（L1/L2）にrejectが無い **かつ** 未解決の構造エラーが無い。
- 包括（excludesFromBilling で行除外＝解決済み）は**ブロックしない**。
- 背反・初再診の競合・不正コード・部位不一致など未解決のエラーは**提出不可**（そのまま出すと返戻のため）。

### 4.4 出力（CalculationResult）

```
lines:       ClaimLine[]（包括除外後の請求行）
issues:      CalculationIssue[]（severity, ruleId, message, procedureCode?, excludesFromBilling?）
totalPoints: 包括除外後の合計点数
```

## 5. 算定支援アラート仕様

### 5.1 3層・非ブロック設計

| レベル | 表示 | 意味 | 発火源 |
|---|---|---|---|
| error | 🔴確認必須 | 客観的に黒（届出状況が分かるとき） | facility_standard（未届で算定） |
| warning | 🟡確認推奨 | 審査裁量だが疑わしい | diagnosis_procedure（病名↔処置の不適応） |
| proposal | 💡取りこぼし提案 | 該当すれば算定できる | age_time_site（通則・注） |

- **ブロックしない**。すべて上書き可能。最終判断は歯科医師。
- **根拠（出典）を必ず併記**（指図でなく参考情報）。
- **既読学習**: 承認したパターン（contextKey = ルールID×病名×処置）は次回から非表示。
- **総量制御**: 1件ずつポップアップせず、レセプト単位で一覧に集約。

### 5.2 精度方針（誤検知ゼロ設計）

- 審査事例の「適応」は“認める一例”であり“必須”ではない → **不適応（forbidden）のみ警告**にする
  （「対応病名なし」は誤検知の温床のため発火させない）。
- 病名は略号・和名（傷病名マスタ35,716件）・コードで解決。解決できない19トークンは
  `unresolvedDiseaseTokens` で可視化（黙って捨てない／docs/curation-backlog.md）。
- 初診料/再診料の年齢・時間外加算は自動算定が担当するため、提案で二重に出さない。
- 同一(処置×病名)の不適応・同一(ルール×病名)は1件に集約。
- 結果: 正常レセプトはアラート0件、誤り（例: 抜髄をPer病名）は1件だけ正しく発火。

## 6. レセプト電算（UKE）出力仕様

### 6.1 ファイル仕様

- 記録条件仕様（歯科用）令和8年6月版に準拠。全16レコード定義。
- 文字コード Shift_JIS、改行 CR+LF、ファイル末尾に EOF（0x1A）。引用符規定に従う。
- ファイル構造: 先頭 UK（受付情報）…末尾 GO（請求書）各1個、レセプトは IR→RE→保険→HS→明細→CO 順。

### 6.2 組立ロジック

- **月次集約**: 同一患者の複数受診を1レセプトに集約。同一（コード・診療識別・加算・点数）の行は
  1つの明細にまとめ、受診日ごとに算定日情報（1〜31日）を積む。算定日合計＝回数。
- **診療識別グループ化**: 明細を診療識別（別表20）昇順に並べ、同一識別の2件目以降は診療識別を省略。
- **診療実日数**: 算定行が生じた日のみ計上（算定ゼロの受診は数えない）。
- レセプト種別（別表6）・負担区分（別表21）・歯式（FDI→6桁）を自動判定。
- 返戻→再請求（別添C）の組立に対応（HI/HG/HR除去、履歴ブロック温存、UK/GO付加）。

### 6.3 提出前自己点検（L1/L2）

支払基金「受付・事務点検ASP」相当の機械点検。
- **L1（1xxx）**: ファイル単位（UK/GO存在・各1個・GO総件数/総点数整合 等）。
- **L2（2xxx）**: レセプト単位（IR→RE順・歯科種別3始まり・保険者番号チェックデジット・
  HS存在・明細存在・算定日合計＝回数・傷病名コード実在 等）。
- 重大度: reject（受付不能相当）/ review（要確認）。reject ゼロで submittable。

### 6.4 確認試験シミュレータ（仮想）

`runKakuninShiken`（src/receipt/submission-sim.ts、`npm run shiken`）。
提出されたUKEバイト列を受付側として復号し直し、受付L1→事務点検L2→合格判定を再現。
判定: 受付不能 / 返戻 / 合格。**形式・構造の確認試験の再現であり、審査委員の医学的判断は再現しない。**

## 8. 電子カルテ保存仕様（電子保存3原則・3省2ガイドライン）

`KarteStore`（src/karte/store.ts）が、`chart.ts` のハッシュ連鎖の上に運用機能を載せる。

| 原則 | 実装 |
|---|---|
| 真正性 | 追記専用・確定済み記録は変更不可・SHA-256ハッシュ連鎖で改ざん検知。訂正は supersedes による新版追加 |
| 見読性 | readableText() で人が読める形に整形（個別指導・監査で提示可） |
| 保存性 | AES-256-GCM 暗号化バックアップの往復（exportBackup/importBackup）＋全体整合性検証 verifyIntegrity() |

3省2の技術的安全管理: 役割別の最小権限アクセス制御（dentist/hygienist/reception/admin）、
改ざん検知付き監査証跡（全操作を記録）。※DB永続化・運用規程・DPIA は別途。
