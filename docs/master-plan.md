# Sourirette 全体設計書（マスタープラン）

レセコン一体型・歯科専用クラウド電子カルテをフルスクラッチで開発するための設計書。
最終更新: 2026-06 / ステータス: Phase 0（基盤設計・骨格実装）

> 本書はエンジニア視点の整理であり、法的助言ではありません。制度・法務に関わる判断は
> 必ず専門家（医療法務・社会保険労務・支払基金等の公式窓口）の確認を受けてください。

---

## 1. ビジョンとスコープ

**「録音 → AI下書き（dentia）→ カルテ確定 → 自動算定 → レセプト提出」を1つの流れにする。**

既存の歯科レセコン/電子カルテは「入力した処置の点数計算」が中心で、診療現場の記録作成
そのものは支援しない。Sourirette は dentia の AI 下書きを入口に、確定カルテと保険請求まで
一体で扱うことを差別化軸とする。

### スコープ（最終形）

- 患者管理（基本情報・保険情報・公費）
- オンライン資格確認（マイナ保険証）連携
- 歯科カルテ（歯式・部位単位の所見/処置・傷病名・SOAP・画像添付）
- 診療報酬の自動算定（歯科点数表準拠、施設基準による分岐、月内回数チェック）
- レセプト電算ファイル（UKE）出力 → オンライン請求
- 窓口会計（患者負担金計算・領収書/明細書発行）
- dentia 連携（AI 下書きの取り込み → 医師確認 → 確定）

### スコープ外（当面）

- 医科診療・調剤
- 予約システム / リコール管理（将来モジュール）
- 介護保険・自費治療の高度な見積管理（最小限のみ）

## 2. 制度・法規制要件（設計の前提条件）

詳細チェックリストは [`regulatory-checklist.md`](./regulatory-checklist.md)。設計に直結する要点のみ：

### 2.1 電子保存の3原則（カルテを電子保存するための条件）

| 原則 | 設計への反映 |
| --- | --- |
| **真正性** | カルテ記録は追記専用（UPDATE/DELETE 禁止をDBトリガで強制）。訂正＝新版の追加。記載者・確定者・確定日時の記録。SHA-256 ハッシュチェーンで改ざん検知 |
| **見読性** | 確定時点の表示内容をそのまま再現できるスナップショット保存。印刷/PDF出力 |
| **保存性** | カルテ完結の日から5年以上の保存（実務上はそれ以上）。バックアップ・リストア手順。フォーマットのマイグレーション計画 |

### 2.2 3省2ガイドライン

- 医療情報システムの安全管理に関するガイドライン（厚労省）＋ 経産省・総務省ガイドライン。
- クラウド提供事業者として：アクセス管理、監査ログ、暗号化（保存時/通信時）、
  リスク分析文書、BCP。**設計段階から監査ログと権限モデルを組み込む**（後付け不可）。

### 2.3 レセプト電算処理システム

- 提出形式は厚労省/支払基金の **「記録条件仕様（歯科用）」** に準拠した UKE ファイル
  （Shift_JIS、固定のレコード識別子を持つ CSV 系フォーマット）。
- 使用するコードは支払基金が公開する **基本マスタ（診療行為・傷病名・医薬品・修飾語・特定器材等）**。
  マスタは毎月更新があり、**2年ごとの診療報酬改定**（直近 2026 年 4 月）で大規模に変わる。
- 提出前に支払基金の **確認試験** を通す必要がある（実質的な参入関門）。

### 2.4 オンライン資格確認・オンライン請求

- 保険医療機関にはオンライン資格確認（マイナ保険証）の導入が原則義務化済み。
  資格確認端末との連携 API（医療機関等向けポータルで仕様公開）への対応が必要。
- オンライン請求はIPsec+IKE もしくは IP-VPN の閉域接続＋電子証明書。

### 2.5 医療機器該当性

- 電子カルテ・レセコン自体は許認可制ではない（プログラム医療機器に非該当の見込み）。
  ただし AI 出力が「診断」に踏み込むと SaMD 該当リスクがあるため、
  **AI は記録作成支援に限定し、確定は医師の操作を必須にする**（dentia と同方針）。

### 2.6 標準化の潮流

- 厚労省は標準型電子カルテ・電子カルテ情報共有サービス（HL7 FHIR ベース）を推進中。
  傷病名等の内部コードは公式マスタ準拠とし、**FHIR へのマッピング層を将来追加できる**
  データモデルにしておく。

## 3. システム構成

```
┌──────────────────────────── クラウド（Sourirette） ───────────────────────────┐
│                                                                              │
│  Web フロント（受付/診察室/会計）                                              │
│        │                                                                     │
│  API サーバ ──┬─ ドメイン層      患者・保険・歯式・カルテ（追記専用）            │
│              ├─ 算定エンジン    マスタ参照・算定ルール・月内回数チェック         │
│              ├─ レセプト出力    UKE 生成・点検（エラー/警告）                   │
│              └─ dentia 連携     AI下書き取り込み（下書き→医師確認→確定）        │
│        │                                                                     │
│  PostgreSQL   業務データ（RLS/トリガで保護）＋ 公式マスタ ＋ 監査ログ            │
└──────────────────────────────────────────────────────────────────────────────┘
         │（閉域接続・電子証明書）                │（院内）
   支払基金/国保連（オンライン請求）        資格確認端末（オン資）
```

### モジュールと依存方向

```
domain（歯式・患者・カルテ） ←─ billing（算定エンジン） ←─ receipt（UKE出力）
        ↑                                ↑
   公式マスタは billing/receipt からのみ参照（domain はマスタ非依存）
```

- **domain**: 制度から独立した不変の概念（歯・部位・記録の版管理）。最も安定。
- **billing**: 点数表のルールを実装。**改定のたびに変わる前提**で、ルールを
  バージョン付きの独立ユニット（`Rule`）に分離し、適用期間で切り替える。
- **receipt**: 記録条件仕様への直列化。仕様改定に追従。

## 4. データモデル（中核）

実装: [`db/schema.sql`](../db/schema.sql)

```
clinics ─< staff（職員・役割）
clinics ─< patients ─< insurance_coverages（保険・公費、期間付き）
patients ─< visits（受診）─< chart_entries（カルテ記録：追記専用・版管理・ハッシュ連鎖）
visits   ─< performed_procedures（診療行為：コード＋部位＋数量）
patients ─< diagnoses（傷病名：開始日・転帰・部位）
*        ─> audit_log（全操作の監査ログ）
m_procedure / m_disease / m_drug（支払基金マスタの取込先、適用期間付き）
```

設計判断：

1. **chart_entries は追記専用。** `status` が `final` になった行は DB トリガで UPDATE/DELETE を
   拒否。訂正は `supersedes_id` で旧版を指す新規行。各行に `content_hash`
   （前行ハッシュ＋本文の SHA-256）を持たせ改ざん検知（真正性）。
2. **部位は FDI（ISO 3950）2桁表記で正規化**（11–18/21–28/31–38/41–48、乳歯 51–55…85）。
   表示は日本式（右上6 等）に変換。レセ電算の部位コードへは出力層で変換。
3. **マスタは適用期間（valid_from / valid_to）付き**で複数世代を共存させる。
   過去日付の診療の再計算・返戻再請求で過去の点数が必要になるため。
4. **保険情報も期間付き**（月途中の保険変更でレセプトが分かれるため）。

## 5. 算定エンジン設計

実装の骨格: [`src/billing/`](../src/billing/)

### 5.1 原則

- **点数・コードのハードコード禁止。** 点数は必ずマスタ（`MasterRepository`）から
  診療日で引く。ルールは「どのコードを、どの条件で算定してよいか」だけを知る。
- ルールは `Rule` インターフェースを実装する小さなユニットの集合。
  各ルールは **適用期間** を持ち、改定時は新ルールを追加して旧ルールを閉じる（削除しない）。
- エンジンの出力は「算定結果（コード・部位・点数）」＋「エラー（算定不可）」＋
  「警告（要確認）」の3層。**勝手に確定しない**。最終確定は人間。

### 5.2 評価コンテキスト

```ts
CalculationContext {
  patient        // 年齢・保険種別
  visit          // 診療日・初診/再診の別
  procedures     // 入力された診療行為（コード・部位・数量）
  diagnoses      // 有効な傷病名
  history        // 同月・同日の算定履歴（回数制限チェック用）
  facility       // 施設基準（届出）の有無
}
```

### 5.3 代表的なルール類型（歯科）

| 類型 | 例 | 実装方針 |
| --- | --- | --- |
| 相互排他 | 同一日の初診料と再診料 | 同一 visit 内のコード集合チェック |
| 回数制限 | 月1回まで／日1回まで | `history` の同月・同日カウント |
| 部位整合 | 抜歯した歯への処置 | 部位ごとの処置履歴・傷病名との突合 |
| 施設基準分岐 | 届出の有無による加算・点数差 | `facility` フラグで分岐 |
| 包括・逓減 | 同時算定で一方が包括される | ルール間の優先順位と打ち消し |
| 病名整合 | 処置に対応する傷病名の存在 | レセプト点検（receipt 層の出力前チェック） |

### 5.4 改定追従の運用

1. 改定告示後、支払基金の新マスタを取込（`valid_from` = 施行日）。
2. 変更された算定ルールを新 `Rule`（適用期間付き）として追加。
3. 新旧の境界日付近のテストケース（ゴールデンテスト）で回帰検証。
4. **このサイクルを2年ごとに必ず回す。** 体制が組めないならこのプロジェクトは成立しない
   （最大のリスク。§8 参照）。

## 6. レセプト出力（UKE）とオンライン請求

実装の骨格: [`src/receipt/`](../src/receipt/)

- 月締めで患者×保険ごとにレセプトを構成し、記録条件仕様（歯科用）のレコード列に直列化。
- 文字コードは Shift_JIS（出力層でエンコード。内部は UTF-8）。
- レコード定義（IR/RE/HO/SY/CO 等のフィールド仕様）は **支払基金公開の公式仕様書から転記して
  確定させる**。リポジトリには枠組み（直列化基盤と型）のみ実装し、推測で埋めない。
- 提出前点検（病名漏れ・回数超過・資格エラー）をレセプト単位で実行し、
  エラーは提出ブロック、警告は確認付き提出とする。
- オンライン請求の接続（閉域網・証明書）は Phase 3 で。それまでは UKE ファイルの
  媒体提出（または既存レセコンへの読込）で検証する。

## 7. ロードマップ

| Phase | 内容 | 完了条件 |
| --- | --- | --- |
| **0（今ここ）** | 設計確定・基盤骨格（ドメイン/算定/出力の枠組み・スキーマ・テスト基盤） | 本書合意・骨格コードがビルド/テスト通過 |
| **1** | カルテ本体：患者・保険・歯式入力・傷病名・追記専用カルテ・監査ログ・dentia 取込 | 1医院で紙/既存システムと並走運用できる（請求は既存レセコン） |
| **2** | 算定エンジン：公式マスタ取込・主要算定ルール・窓口会計（負担金計算） | 実際の1ヶ月分の診療で既存レセコンと算定結果が一致（差分ゼロを目標にした突合） |
| **3** | レセプト：UKE 出力・提出前点検・支払基金の確認試験・オンライン請求接続 | 確認試験パス → 実レセプト提出で返戻ゼロの月を作る |
| **4** | 一体運用：オン資連携・施設基準管理・複数医院展開・3省2GL 監査対応 | 外部医院への商用提供 |

Phase 2 の「既存レセコンとの突合」が品質担保の核。**正解データ（既存レセコンの算定結果）
と突き合わせて差分を潰す**進め方を取る。

## 8. リスクと前提（正直な評価）

1. **改定追従が最大の継続コスト。** 2年ごとの点数表改定＋毎月のマスタ更新に追従する
   体制を維持できることが事業の前提。個人開発で最も折れやすいポイント。
2. **歯科には ORCA に相当する無料レセコンが存在しない**ため、算定エンジンは完全自作になる。
   逆に言えば、ここを作り切れば参入障壁そのものになる。
3. **確認試験・オン資・閉域網接続**は書類と手続きの時間がかかる（数ヶ月単位）。
   Phase 3 開始時点で早めに着手する。
4. 算定ルールの細部（施設基準・地域差・解釈通知）は公式資料＋実際の歯科医院の運用知識が
   必須。**協力してくれる歯科医院（dentia のβ医院）を Phase 1 から巻き込む**。
5. 本書内の制度記述は実装時に必ず一次資料（厚労省告示・支払基金仕様書）で裏取りする。
