結論（保存先： docs/specs/uke-examples-index.md ）：参照すべき記録例の本体は支払基金が公開する単一PDF jiki_s01.pdf （「電子

3. 記録の留意事項の所在（HS部位記録／SS数量・点数）

3-1. HS（傷病名部位レコード）の歯式・部位記録 — 第8部 p22〜37
  レコード識別子は歯科では「HS」（医科の傷病名レコード「SY」とは別。これはゴールデンテストの分岐で要注意）。

  歯式コードは**「歯種コード4桁＋状態コード1桁＋部分コード1桁＝計6桁」**で1歯を表し、HSレコードの「歯式コード（傷病名）」「歯
  式コード（補足コメント）」項目に記録。**最大64個（=384バイト、6の倍数）**まで連記。                Ssk
  配列順の留意事項（記録条件仕様の注記＝手引きも同準拠）：複数歯は「右上遠心→右上近心→左上近心→左上遠心→右下遠心→右下近心→
  左下近心→左下遠心」の順。乳歯・過剰歯が永久歯と同部位に並存する場合は2つの歯式を同じ配列順で並記。                                   ssk   ブロック単位を表す特
  別な歯式コードは単独（6桁のみ）で記録。

  HSレコードは診療開始日の昇順で記録。       Ssk   要点・表示ルールは第8部「3 入院外レセプト表示について」p25。


3-2. SS（歯科診療行為レコード）の数量・点数記録 — 第9部 p38〜45 ＋ 第14部 p65〜78
  SSレコードの項目構成（記録条件仕様準拠）： レコード識別情報(SS) / 診療識別 / 負担区分 / 診療行為コード / 診療行為数量データ1・2
  / 加算コード1〜35＋各加算数量データ / 点数 / 回数 / 算定日情報(1〜31日) 。

  歯科の加算は医科と異なり、加算先と同一レコード内の「加算コード1〜35」に注加算・通則加算・基本加算・材料1加算・材料2加算の順
  で記録（医科のように別レコードにしない）。記録順と点数計算は第9部「3 加算の記録順及び点数計算」p43。

  数量データは歯科診療行為マスター・基本テーブルの「きざみテーブル関連識別」に1/2が設定された行為に整数で記録。

  共通記録方法（複数レコードにまたがる留意）は第14部「1 各レコード項目の記録」：診療識別の記録 p65、負担区分の記録 p68、点数の
  記録 p70、レセプト表示方法 p71〜78。

  SSレコードには歯式コード項目は存在しない。診療行為に紐づく歯式が必要な場合は**COレコード（歯式コード（コメント）、最大64
  個・6の倍数バイト）**で表現する設計。これは出力エンジン実装で間違えやすいポイント。

 補足（要確認）：上記の項目順・加算の同一レコード内記録・歯式の格納先（HS/CO）は記録条件仕様（歯科用）に基づく確定情報。ただ
 し別表15〜17（歯種/状態/部分コード）は現行仕様で「削除」となっており、歯式は「別に定める歯式コード（1コード6バイト）」に一本化
 されているとの調査結果がある。歯式コードの最新の定義は**マスターファイル仕様説明書（歯式マスター）と記録条件仕様（令和8年6月
 版）**で必ず突き合わせること。




4. UKE出力検証ツール（ゴールデンテストの補完・代替）

4-1. 支払基金「受付・事務点検ASP」（本命）
  オンライン請求システム内で、請求確定前にレセプトデータを送信して事務点検チェックを受けられる機能。「ASPあり」で送信するとエ
  ラーが返り、当月内（12日まで）に修正・再請求できる。

  チェック内容を公開資料から把握できるのがゴールデンテスト設計上の価値：

    形式・コード存在・記録順序・レコード間関連のチェックロジックは「受付・事務点検ASPに係るチェックロジック（歯科）令和6年7
    月版」（ checklogic_shika.pdf ）に「第2 基本チェック」「第3 項目別チェック（受付情報/医療機関情報/レセプト共通/保険者/公
    費/資格確認/受診日等/窓口負担額/傷病名部位…）」        Ssk   として記載。

    個別エラーと受付不能/要確認の区分は「受付・事務点検ASPに係るチェック一覧（歯科）令和6年11月版」
    （ checkichiran_shika.pdf ）。エラーコード体系：L1（1000番台＝医療機関単位、要再請求）/ L2（2000番台＝レセプト単位、要
    再請求）/ L3（3000番台＝要確認）/ L4・L48（4000・4800番台＝要確認・正常処理）。「                    Ssk   受付不能ASP」欄に「●」がある
    ものは医療機関の修正必須。歯科特有チェック例：レセプト種別1桁目が「3（歯科）」以外／傷病名部位レコードが1レセプト100件以
    上／医療機関情報レコードの次がレセプト共通レコードでない、等。                 Ssk   Miyagi-kokuho


  一覧の掲載場所：オンライン請求システムの「マニュアル▶その他▶受付・事務点検ASPに係るチェック一覧（歯科）」、または支払基金
  HP「トップ▶診療報酬の請求支払▶オンライン請求」。上記直リンクからも取得可。


4-2. レセプトデータの確認試験（オンライン確認試験）
  正式名称は**「レセプトデータの確認試験」**。本請求前にレセプトデータが記録条件仕様等に適合しているかを確認する仕組みで、確認
  試験可能期間内なら何回でも実施でき、前月診療分までのレセプトデータを対象にできる（                          Osakakokuhoren   出典：支払基金/国保連合会
  「オンライン請求システム操作手順書」第10章「レセプトデータの確認試験を実施する」）。確認試験のため実際には請求されない／支払
  基金のみ提供。エラーメッセージは受付・事務点検ASP結果リストと同じコード体系（「受付処理結果リスト、受付・事務点検ASP結果リ
  スト及びオンライン確認試験結果リストに出力されるメッセージ」              Ssk   Ssk   と各一覧に明記）。


4-3. 国保中央会側の同種ツール
  国民健康保険中央会も同じオンライン請求システム基盤を共用し、「受付・事務点検ASP」「確認試験」を提供。**ベンダー向けには「レ
  セプト電算処理システムに係る接続試験実施要領（歯科）」（PDF約594KB）と「光ディスクを用いた費用の請求に係る接続試験依頼書
  （歯科）」**を公開（     Kokuho   kokuho.or.jp/system/resept/ ）。チェック一覧（歯科）は国保連合会版も各県サイトに掲示（例：宮
  城県国保連 令和4年5月版）。


4-4. ORCA（参照実装）
  日医標準レセプトソフト（日レセ／ORCA, Online Receipt Computer Advantage）。日本医師会が2002年から「日医オープンソース
  使用許諾契約」に則りプログラムを無償公開している                   Med   オープンソースのレセコンで、レセプト電算（レセ電）データ出力機能を標準
  装備。歯科版もある。UKE出力の挙動を比較する参照実装として有用だが、周辺サービス（マスタ更新・サポート等）は商用版契約が前提
  な点に注意。ソースとマニュアル類は orca.med.or.jp および FTP（ ftp.orca.med.or.jp ）から入手可能。
